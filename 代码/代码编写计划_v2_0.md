# 一、技术选型

## 1.1 为什么是 HTML5 Canvas 而不是 Unity/Godot？

| 维度 | Canvas+TS | Unity | Godot |
|------|:--:|:--:|:--:|
| AI 代码生成质量 | ⭐⭐⭐⭐⭐（DeepSeek 对 TS 极强） | ⭐⭐（C# 中等） | ⭐⭐⭐（GDScript 可以） |
| 迭代速度 | 即时预览，无编译 | 中等 | 快 |
| 对 Gorogoa-like 的适配 | 天然——图层=Canvas层 | 需要2D工具链 | 需要2D工具链 |
| 部署 | URL 即游戏 | 需要打包 | 需要打包 |
| 包体 | 按需加载图片 | 整包 | 整包 |
| 学习曲线 | 低（就是画图+拖拽） | 高 | 中 |

**结论：** Canvas 对"图片拼接解谜"这个品类是天然匹配——不需要物理引擎、不需要3D、不需要复杂的场景管理。所有交互本质是"拖拽画面格→对齐判断→状态切换"。DeepSeek 对 TypeScript 的代码生成质量远高于 C# 和 GDScript。

## 1.2 技术栈明细

| 层 | 技术 | 用途 |
|------|------|------|
| 语言 | TypeScript 5.x | 类型安全，AI 友好的代码结构 |
| 渲染 | Canvas 2D API | 多层画布：背景层/谜题层/VFX层/UI层 |
| 构建 | Vite | 极速 HMR，零配置 TS |
| 状态管理 | 自研 FSM | 无框架依赖，每个谜题一个状态机 |
| 存储 | localStorage | 存档（当前章+盖章状态） |
| 视频 | `<video>` 叠加层 | Seedance VFX 触发 |
| 音频 | Web Audio API | 铜声精准触发+环境音 |
| 测试 | Vitest | 谜题逻辑单元测试 |
| 部署 | Vercel/Netlify | 免费托管，自动 HTTPS |

# 二、核心系统设计

## 2.1 Canvas 多层架构

```
┌─────────────────────────┐
│   UI Canvas (盖章/释义)   │ ← 最上层，始终在最前
├─────────────────────────┤
│   VFX Canvas (视频/粒子)  │ ← Seedance 触发时覆盖
├─────────────────────────┤
│   Puzzle Canvas (拖拽层)  │ ← 玩家交互层
├─────────────────────────┤
│   BG Canvas (背景层)     │ ← 静态底图
├─────────────────────────┤
│   Map Canvas (地图层)    │ ← 序章/尾声叠加
└─────────────────────────┘
```

每个 Canvas 独立渲染，引擎统一管理合成顺序。拖拽事件在 Puzzle Canvas 上捕获，对齐判定后通知 BG Canvas 更新。

## 2.2 谜题基类

```typescript
// PuzzleBase.ts
interface PuzzleConfig {
  id: string;
  type: 'alignment' | 'nest' | 'rotation' | 'sequence' | 'direction';
  frames: PuzzleFrame[];       // 关键帧
  snapThreshold: number;       // 吸附阈值(px)
  progressiveFeedback: {       // 渐进反馈
    near: string;              // 靠近正确位置→什么画面变化
    match: string;             // 匹配成功→什么画面变化
    fail: string;              // 完全不对→什么画面变化
  };
}

abstract class PuzzleBase {
  abstract checkAlignment(): AlignmentResult;
  abstract getProgressFeedback(): FeedbackFrame;
  abstract onSolve(): void;
  abstract onReset(): void;
}
```

## 2.3 状态机设计

```
[章初始] → [谜题1未解] → [谜题1已解] → [盖章动画] → [释义弹出]
    ↓                          ↓
[翻页过渡] ← [章内所有谜题已解] ← [章内其他谜题...]
    ↓
[下一章]
```

每个谜题独立 FSM：`IDLE → DRAGGING → NEAR → SNAPPED → SOLVED`。

## 2.4 交互范式

全作仅三种交互原语：

| 交互 | 实现 | 使用位置 |
|------|------|------|
| **拖拽对位** | mousedown/touchstart → mousemove → 距目标<threshold → snap | 陶范/纸成/托/衣归/孔老/云想/归田/烬(烟雾) |
| **拖拽嵌套** | 平移+缩放画面层，检测洞口→目标覆盖 | 灰页 |
| **旋转匹配** | 单指旋转（弧形拖拽），三档吸附 | 瘦骨（唯一异类）+ 烬(风铎) |

# 三、DeepSeek vs 千问 分工策略

## 3.1 分工原则

| 能力维度 | DeepSeek | 千问(Qwen) | 策略 |
|------|:--:|:--:|------|
| TypeScript 代码生成 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | **DeepSeek 承担全部代码** |
| 算法/数学/状态机 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | **DeepSeek** |
| 中文注释/文档 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 千问优化中文文档 |
| 视觉理解/图片验证 | ⭐⭐(V3无视觉) | ⭐⭐⭐⭐⭐(Qwen-VL) | **千问做全部视觉验证** |
| 代码审查 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 两者交叉审查 |
| 调试/错误分析 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | DeepSeek 主导 |
| 资产生成 Prompt 撰写 | ⭐⭐⭐ | ⭐⭐⭐⭐ | 千问（中文Prompt更强） |

## 3.2 具体分工表

### DeepSeek 负责（代码生成 · 逻辑 · 算法）

| 模块 | 文件 | 关键难点 |
|------|------|------|
| **Canvas 多层管理** | `CanvasManager.ts` | 合成顺序、性能优化 |
| **拖拽系统** | `DragHandler.ts` | 鼠标+触摸双轨、惯性、吸附 |
| **对位检测** | `HitDetector.ts` | 多边形碰撞、旋转后矩形检测、阈值自适应 |
| **补间动画** | `AnimationEngine.ts` | 缓动函数库、关键帧插值 |
| **5 类谜题逻辑** | `AlignmentPuzzle.ts` `NestPuzzle.ts` `RotationPuzzle.ts` `SequencePuzzle.ts` `DirectionPuzzle.ts` | 每类不同的对齐判定算法 |
| **章节状态机** | `ChapterFSM.ts` | 章间跳转、暗页过渡、翻页动画 |
| **存档系统** | `SaveManager.ts` | 盖章状态序列化、中断恢复 |
| **音频触发** | `AudioManager.ts` `BronzeSound.ts` | 铜声11声的精准时序触发 |
| **UI 特效** | `StampEffect.ts` `DefinitionPopup.ts` | 盖章粒子和释义动画 |
| **构建配置** | `vite.config.ts` `tsconfig.json` | Vite + TS 最佳配置 |
| **单元测试** | `alignment.test.ts` `rotation.test.ts` 等 | 核心对位逻辑的数学验证 |

#### DeepSeek 使用方式

1. **架构设计对话**：先和 DeepSeek 讨论模块接口/数据流/状态管理方案，产出详细伪代码
2. **逐模块生成**：将 DeepSeek 输出的伪代码→TypeScript 实现→粘贴到项目中
3. **增量迭代**：每个模块在 Vite HMR 下实时验证→发现问题→继续和 DeepSeek 讨论修复
4. **关键算法单独验证**：对齐检测的数学逻辑先用 DeepSeek 生成单元测试→通过后再集成

### 千问(Qwen-VL)负责（视觉验证 · 资产审查 · Prompt优化）

| 任务 | 具体内容 | 千问能力 |
|------|------|:--:|
| **资产质量审查** | 将 Seedream 生成的 235 张图依次交给千问 VL 检查：是否符合场景描述？颜色是否正确？元素是否缺失？ | Qwen-VL 最强 |
| **谜题对齐验证** | 截图游戏运行状态→千问 VL 判断：两格是否视觉上真的对齐了？吸附感是否自然？ | 视觉判断 |
| **颜色体系一致性** | 检查每章画面是否符合颜色语义（朱红/青铜青/赤红等） | 色彩识别 |
| **人物侧写检查** | 检查 22 张人物图是否满足"侧写不告"原则（不出现面部、不标注姓名） | 面部检测 |
| **即梦 Prompt 撰写** | 为 235 张图撰写 Seedream Prompt（中文），确保生成质量 | 千问中文Prompt强 |
| **Seedance Prompt 撰写** | 为 16 段视频撰写 Seedance Prompt（动作描述/风格/时长） | 千问中文Prompt强 |
| **交互 UI 审查** | 检查教学关引导是否清晰/盖章动画是否自然/释义弹出是否易读 | 视觉理解 |
| **跨章视觉锚审查** | 检查手印三部曲/水的三种面貌/牡丹六环的视觉一致性 | 跨画面对比 |

#### 千问使用方式

1. **Prompt 撰写阶段**：将所有场景描述喂给千问→产出 Seedream/Seedance Prompt→人工微调→批量生成
2. **质量审查流水线**：生成一批图→千问 VL 逐张审查→标注通过/需重生成/需微调
3. **运行时截图验证**：游戏运行后截图→千问 VL 判断"这个画面是否符合设计意图？"
4. **代码+视觉联动**：DeepSeek 写的代码生成的画面→千问 VL 检查"颜色/位置/对齐是否符合预期"→反馈给 DeepSeek 调整

### 两者协作流水线

```
[千问] 撰写 Seedream Prompt
    ↓
[Seedream] 批量生成资产
    ↓
[千问 VL] 逐张审查 → 不通过 → [千问] 改写 Prompt → 重新生成
    ↓ 通过
[DeepSeek] 编写模块代码 + Vite HMR 实时验证
    ↓
[千问 VL] 运行时截图验证 → 视觉问题 → [DeepSeek] 调整参数
    ↓ 通过
[交付]
```

# 七、代码规范

1. **所有谜题类继承 PuzzleBase**，不在章节类中直接写谜题逻辑
2. **整型常量化阈值**：`SNAP_THRESHOLD = 30` 而非魔法数字
3. **帧动画用 async/await**：`await anim.play(frame1, 300)` 
4. **音效触发用事件总线**：`EventBus.emit('bronze:sound', 'guDu')`
5. **资产路径用 AssetManifest 映射**：`ASSETS.bg.erlitou.taoFan` 而非硬编码路径
6. **每个章节一个 Chapter 子类**：`Chapter01_Erlitou extends ChapterBase`
7. **单元测试覆盖所有 Puzzle 类的 checkAlignment 方法**
